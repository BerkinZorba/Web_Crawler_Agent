# Production evolution (recommendation)

A production system would split the monolithic coordinator into **separate, scalable services**: crawl workers (possibly many processes or containers) fed by a **durable work queue** (e.g. Redis, Kafka, or cloud-native equivalents), with explicit **rate limiting**, **`robots.txt` and crawl-delay compliance**, per-host politeness, and optional **headless/JS rendering** only where justified. The single-threaded SQLite writer would give way to **concurrent fetchers** and a **clear API boundary** between “discover/fetch/normalize,” “index,” and “query,” plus authentication, configuration management, and network egress controls appropriate to a datacenter or managed cloud environment.

On the data side, **SQLite and hand-rolled term tables** would not carry large-scale search: you would typically move to a **managed search engine** (OpenSearch, Elasticsearch, or similar) or a database with **full-text search and ranking** tuned for your traffic, possibly augmented with **embeddings** for semantic recall if product requirements demand it. Operationally, the student version omits **metrics, tracing, structured logging, alerting, crawl budgets, data retention and legal review, secret handling, backup/restore, and incident runbooks**—all of which become mandatory once crawling and search are exposed beyond a single developer machine.
